## 1. 产品定位

**一句话**：个人健身训练记录工具，在复健期辅助训练执行与复盘。离线、单文件、移动端优先。

**核心转变（v3.0）**：从"计划展示器"变为"训练引导器"——用户不必按计划顺序做，App 跟随用户实际训练节奏，逐动作专注完成，自动记录执行顺序以供分析。

---

## 2. 用户角色

单人使用，无多角色。用户即训练者，也是数据分析者。

## 4. 跨功能规则

### 4.1 完成状态规则

**一个动作的完整生命周期：从"未开始"到"已归档"**

| 状态 | 列表页显示 | 大卡片 | 可操作 |
|------|-----------|--------|--------|
| 未开始 | 空心圆圈 | 可进入，所有组 undone | 做组、调重、写备注 |
| 进行中 | 橙色半透明圆圈 + `⋯` | 部分组 done，可继续 | 同上 |
| 已完成 | 绿色实心 ✓ | 可重入查看，只读 | 查看数据与备注，按钮为"关闭" |
| 已归档 | 不展示 | — | 历史中只读 |

- **进行中 vs 未开始的区分**：只要任意一组 done，状态即变为"进行中"。这是新引入的状态，v2.0 无此区分。
- **已完成动作可重入查看**：点进去只读模式，组表格不可操作，备注 readonly，按钮变为"关闭"。
- **重置动作**：大卡片底部"× 重置"按钮，将当前动作所有组 undone、清除备注。

### 4.2 备注与数据的分离

- **备注是文本**，记录在 `log.note` 中，与重量/次数数据一同归档
- **重量/次数是数据**，用于分析和渐进对比
- 两者不交叉：备注不影响任何计算或图表

### 4.3 计时器规则

| 动作 | 组间休息 |
|------|---------|
| 臀推/哈克深蹲/坐姿划船/Vertical Row 等大重量复合 | 90s |
| 辅助引体向上 | 120s |
| 其他所有 | 60s |

规则与 v2.0 一致，无需变更。

## 5. 数据模型

### 5.1 核心变更（相对 v2.0）

```typescript
// ExerciseLog — 新增字段
interface ExerciseLog {
  sets: SetRecord[]
  completed: boolean
  note: string           // <-- 新增：动作级备注
  orderIndex: number     // <-- 新增：当日完成顺序（0 为第一个做的）
}
```

### 5.2 HistoryRecord 变更

```typescript
// HistoryRecord.details 条目 — 增加 note 和 orderIndex
interface HistoryDetail {
  name: string
  target: string
  unit: string
  sets: { weight: number, reps: number }[]
  note?: string           // <-- 新增：归档时从 log.note 带入
  orderIndex: number      // <-- 新增：归档时从 log.orderIndex 带入
}
```

### 5.3 新增：额外动作的数据存储

```typescript
// 额外动作在 logs 中以 extra-{timestamp} 为 key
// 元数据存储在 log 内部，用于 getExById 重建动作信息
interface ExtraExerciseLog extends ExerciseLog {
  _name: string        // 动作名称
  _target: string      // 靶肌肉
  _totalSets: number   // 总组数
  _reps: number        // 每组次数
  _unit: string        // 单位 'lb' | 'kg'
}
```

### 5.4 不变的数据结构

- `AppState`（currentDay, history, logs, weights, cycle 参数, **dataVersion: 3**）——新增 `dataVersion` 字段
- `HistoryRecord`（date, dayName, dur/h/cal, durC/hC/calC, totalSets, details）——保持 v2.0 结构
- `SetRecord`（weight, reps, done）——保持 v2.0
- `DAYS` 数组（三天计划数据）——保持 v2.0

### 5.5 版本兼容

- v3.0 读取 v2.0 的 localStorage 数据时，`logs` 中不带 `note` 和 `orderIndex` 的会被初始化为 `''` 和 `-1`
- `dataVersion: 3` 标记，用于未来迁移

## 7. 存储策略

- **存储键**：`tr_app_v3`（单人使用，无需兼容旧 key 污染问题）
- **迁移**：首次启动检测 `tr_app_v2` 存在 → 自动读取并写入 `tr_app_v3`，完成后清除 `tr_app_v2`。**无提示、无手动导入**——单人工具不需要迁移弹窗
- **数据位置**：localStorage（不变）

## 9. 未决问题

| 问题 | 状态 | 备注 |
|------|------|------|
| v2.0 数据迁移 | **决定：自动迁移，无提示** | 单人工具，检测到 `tr_app_v2` 直接读并写入 v3 key，旧 key 清除。不弹窗 |
| 已完成动作重入查看 | **决定：可以，只读** | 点进去可查看数据和备注，组表格和备注为只读，按钮变为"关闭" |
| "完成动作"后跳转 | **决定：直接退回列表** | 不提示推荐动作 |
| 添加额外动作的表单字段 | **决定：名称 + 靶肌肉 + 组数 + 默认重量 + 单位** | 靶肌肉为自由文本输入，不限制专业名称（填"臀""大腿""背"等即可） |
| 额外动作持久化 | **决定：不持久化到 DAYS** | 仅在当日 logs 中存在，切换天数时自动清除（不提示，单人知道自己在做什么） |
| 额外动作删除 | **决定：列表页额外动作行右侧显示 ✕ 按钮** | 点即删，无二次确认（单人误删也不影响别人） |
| 额外动作计时器 | **决定：默认 60s** | ID 格式 `extra-{timestamp}` |
| 热身/有氧/拉伸是否需要大卡片 | **决定：不需要** | 辅助信息，展开查看即可 |
| 纯有氧日归档的 dayId 值 | **决定：0** | 前端区分用，不显示给用户 |
